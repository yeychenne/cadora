# The deliverable pack — user manual

An evidence pack is the checksummed proof a run happened the way it says — built for a **verifier**
who recomputes hashes. But the person who signs off on the work is rarely the person who recomputes
hashes. The **deliverable pack** is the other half: `cadora deliverable <run-id>` turns one archived
run into a client-facing **delivery report** — the readable story that sits on top of the same
tamper-evident evidence. Same source of truth, same honesty contract, different reader.

---

## 1. What the deliverable is, and why

When a run is archived, `cadora deliverable <run-id>` writes a narrative delivery report — by default
`deliverable.md`, beside the run:

| File | What it is |
|---|---|
| `deliverable.md` | The client-facing delivery report — executive summary, what was delivered per AI-DLC phase, quality & assurance, cost (FinOps), and an evidence appendix. Dependency-free Markdown; renders anywhere. |
| `deliverable.docx` | The same report as a Word document. Optional — written only with `--docx`, and only if the `cadora[deliverable]` extra is installed. |

The deliverable **reuses** the evidence pack's structured report (`build_report`) rather than
re-deriving anything. The evidence pack (`cadora report`) is the checksummed proof; the deliverable
is the readable story on top of it. One source, so the two can never disagree — the story never
outruns the proof.

> **Honesty contract (inherited from the evidence pack).** Every claim in the report is exactly what
> the run archive recorded — deterministic gate verdicts, integrity findings, human-review decisions,
> and per-node cost (backend-reported dollars, *or* a price-table estimate, explicitly flagged with
> `*`). It is **checksummed, not signed**, and it makes no claim about agent honesty beyond what the
> gates and integrity checks verified.

---

## 2. Produce the report — `cadora deliverable`

```bash
cadora deliverable pr1-verify --archive-dir runs
```

```
delivery pack for pr1-verify:
  md        runs/pr1-verify/deliverable.md
```

The report lands **beside the run** by default — `<archive-dir>/<run-id>/deliverable.md` — not inside
the evidence pack's `report/` directory. The deliverable is a sibling of the proof, not part of it.
To put it somewhere else (say, a directory you'll hand to a client), pass `--out`:

```bash
cadora deliverable pr1-verify --archive-dir runs --out ./client-pack
```

```
delivery pack for pr1-verify:
  md        client-pack/deliverable.md
```

Re-running `deliverable` regenerates the report from the current archive, so it always reflects the
run as it stands now.

---

## 3. What's in the report

The report is always the same five sections, in this order — the exact `##` headings emitted by
`render_deliverable`:

1. **`## Executive summary`** — one paragraph: the topology, node count, backends, outcome, total
   cost (estimates flagged), the gate roll-up, and the integrity and human-review counts.
2. **`## What was delivered`** — a per-node table across the AI-DLC phases:
   `Node · Backend · Model · Outcome · Gate · Cost`. An estimated cost carries a `*`, footnoted below
   the table.
3. **`## Quality & assurance`** — gate verdicts, integrity findings across N nodes, recorded
   human-review decisions, and the pointer to the checksummed evidence pack.
4. **`## Cost (FinOps)`** — the total across all nodes, with the number of price-table estimates
   called out.
5. **`## Evidence appendix`** — the archived-file count (each SHA-256 checksummed) and the honesty
   contract restated: checksummed, not signed.

### Worked example

For a small three-node `aidlc-hitl` run, `deliverable.md` reads:

```markdown
# Solution Delivery Report — aidlc-hitl

*Run `pr1-verify` · generated by Cadora 0.10.0 · 2026-07-17T14:20:03+00:00*

## Executive summary

Cadora conducted an AI-DLC pipeline (**aidlc-hitl**) across **3 nodes** on **claude**. Outcome: **✅ PASS**. Total metered cost **$0.1873** (1 node(s) priced from the public price table). Gate verdicts — passed: 1. Integrity findings: 0. Human-review decisions: 2.

## What was delivered

| Node | Backend | Model | Outcome | Gate | Cost |
|------|---------|-------|---------|------|------|
| requirements | claude | claude-opus-4-8 | ✓ | — | $0.0642 |
| design | claude | claude-opus-4-8 | ✓ | — | $0.0511 |
| construction | claude | claude-sonnet-4-5 | ✓ | passed | $0.0720* |

<sub>`*` = cost estimated from the public price table (backend reported tokens, not dollars).</sub>

## Quality & assurance

- **Gate verdicts:** passed: 1
- **Integrity:** 0 finding(s) across 0 node(s)
- **Human review:** 2 recorded decision(s)
- **Verifiable evidence:** every claim above is checksummed — run `cadora report pr1-verify` for the portable, SHA-256-verified evidence pack.

## Cost (FinOps)

Total: **$0.1873** across 3 nodes, 1 estimated from the price table.

## Evidence appendix

- 23 archived file(s), each SHA-256 checksummed.
- Honesty contract: this report claims exactly what the run archive recorded (gate verdicts, integrity findings, human-review decisions, per-node cost); it is checksummed, not signed, and makes no claim about agent honesty beyond what the gates and integrity checks verified.
```

Read it as a client would: the outcome and cost up top, the per-node breakdown next, the assurance
counts, then the appendix that tells them where the proof lives. The `*` on the construction node is
load-bearing — that dollar figure is a price-table estimate (the backend reported tokens, not
dollars), and the report says so rather than presenting it as a metered number.

## 4. The optional `.docx`

Some stakeholders want a Word document. `--docx` renders `deliverable.docx` alongside the Markdown:

```bash
pip install 'cadora[deliverable]'
cadora deliverable pr1-verify --docx --out ./client-pack
```

```
delivery pack for pr1-verify:
  md        client-pack/deliverable.md
  docx      client-pack/deliverable.docx
```

The `.docx` renderer is **soft-imported** — the Markdown core has no dependency on it, so the base
install stays lean. If you pass `--docx` without the extra installed, the command stops clean with a
pointer to the fix:

```
cadora deliverable --docx requires python-docx: pip install 'cadora[deliverable]'
```

The Word document carries the report title, the executive summary, and the delivery table — the same
evidence, in the format a client's document folder expects.

## 5. Proof and story — how the deliverable relates to the evidence pack

The deliverable and the evidence pack are two surfaces over **one** archived run:

- **Evidence pack** (`cadora report` → `cadora sign` → `cadora verify`) — `report.html`,
  `report.json`, `checksums.txt`, and an optional detached signature. The checksummed, tamper-evident
  **proof** a verifier recomputes offline.
- **Delivery report** (`cadora deliverable`) — `deliverable.md` (and optionally `.docx`). The readable
  **story** a stakeholder reads and signs off on.

Both are rendered from the same structured evidence (`build_report`), so the story is always
consistent with the proof. When a stakeholder wants to go deeper, the report's **Evidence appendix**
points them straight at the pack — run `cadora report <run-id>` and hand over the checksummed bundle.
The deliverable never restates or re-documents the pack; it cross-references it.

For everything about producing, signing, verifying, and handing off the evidence pack, see the
**evidence pack** manual — this deliverable sits on top of it and inherits its honesty contract.

---

## 6. Who reads what

| | Evidence pack | Delivery report |
|---|---|---|
| **Reader** | A verifier | A stakeholder |
| **Wants** | Proof nothing changed | The outcome, cost, and assurance in plain language |
| **Does** | Recomputes every hash, checks the signature — offline | Reads the narrative; signs off |
| **Command** | `cadora verify <run-id>` | `open deliverable.md` / `.docx` |
| **Files** | `report.html · report.json · checksums.txt · checksums.txt.sig` | `deliverable.md · deliverable.docx` |

Don't make the stakeholder recompute hashes, and don't make the verifier read prose. Same run, two
surfaces, two desks.

## 7. Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| `no such run 'pr1-verify' in runs/` | No `manifest.json` for that run under `--archive-dir` | Check the run id; point `--archive-dir` at the directory that holds the run |
| `cadora deliverable --docx requires python-docx: pip install 'cadora[deliverable]'` | `--docx` was passed but the optional extra isn't installed | `pip install 'cadora[deliverable]'`, or drop `--docx` and use the Markdown report |
| Report written to `runs/<id>/` when you expected `report/` | Default output is **beside** the run, not inside the evidence pack | Pass `--out <dir>` to place it where you want; the deliverable is a sibling of the pack by design |
| A cost shows a trailing `*` | That node's cost is a price-table estimate (backend reported tokens, not dollars) | Nothing to fix — the `*` is the honesty flag; the footnote explains it |
| Numbers look stale after a re-run | `deliverable.md` reflects the archive at generation time | Re-run `cadora deliverable <run-id>` to regenerate from the current archive |
| Stakeholder asks to verify the claims | The deliverable is checksummed, not signed | Point them to the evidence pack: `cadora report <run-id>`, then `cadora verify` (see the evidence pack manual) |

## 8. Reference

**`cadora deliverable <run-id>`** — write a client-facing delivery report for one run
(markdown; `--docx` optional).
`--archive-dir <dir>` (default `runs`) ·
`--out <dir>` (default `<archive-dir>/<run-id>/`) ·
`--docx` (also render `.docx`; needs `pip install 'cadora[deliverable]'`)

**Writes:** `deliverable.md` — and, with `--docx`, `deliverable.docx` — into the out dir.
Terminal receipt: `delivery pack for <run-id>:` followed by one `md` (and optionally `docx`) line per
file written.

**Sections** (exact `##` headings, in order): `## Executive summary` · `## What was delivered` ·
`## Quality & assurance` · `## Cost (FinOps)` · `## Evidence appendix`. Top heading:
`# Solution Delivery Report — <topology>`.

**Relationship:** reuses `build_report` (the evidence pack's structured report) as its single source.
The pack is the checksummed proof; the deliverable is the readable story on top. Estimates are flagged
`*`; the report is checksummed, not signed. To verify the underlying claims, produce and check the
evidence pack — `cadora report <run-id>`, then `cadora verify <run-id>`.
